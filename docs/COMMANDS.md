# Premiere Bridge panel — command reference

The CEP panel bundled in this package (`src/premiere_bridge/panel/`)
displays log messages from automation scripts AND executes
ExtendScript-backed commands against the open Premiere Pro project. This
document is the full HTTP/command reference; the `premiere-cli` tool is a
thin wrapper over the `/command` endpoint described here.

## Install

    premiere-cli install-panel      # copies the panel + enables PlayerDebugMode
    # restart Premiere Pro, then: Window > Extensions > Premiere Bridge
    premiere-cli doctor             # verify

## How it works

While the panel is open, it runs a small HTTP server on
`http://127.0.0.1:47823`. Send it a log message:

    curl -X POST http://127.0.0.1:47823/log \
      -H 'Content-Type: application/json' \
      -d '{"message": "hello", "level": "info"}'

Or from Python, use the `premiere-log` CLI:

    premiere-log "hello" --level info

`GET /ping` returns `{"ok": true, "panelVersion": "..."}` — used by
`premiere-cli doctor` to detect a stale installed panel.

### Message schema

    POST /log
    {
      "message": "<required string>",
      "level": "info" | "warn" | "error",   // optional, default "info"
      "source": "<optional string>"
    }

Responses: `200 {"ok": true}` on success, `400 {"ok": false, "error": "..."}`
if the body is malformed.

### Commands

The panel also runs ExtendScript-backed commands via `POST /command`:

    curl -X POST http://127.0.0.1:47823/command \
      -H 'Content-Type: application/json' \
      -d '{"command": "get-project-info"}'

Or from Python:

    premiere-cli get-project-info

**Command schema:**

    POST /command
    {
      "command": "<required string>",
      "args": {}                          // optional, command-specific
    }

Responses: `200 {"ok": true, "result": <command-specific>}` on success,
`200 {"ok": false, "error": "<reason>"}` if the command itself failed (e.g.
no project open), `400 {"ok": false, "error": "<reason>"}` if the request
was malformed or named an unknown command. Every command run is also logged
as one line in the panel's log view.

| `export-frame` | `{"outputPath", "timecode", "sequenceName"?}` | `{"sequenceName", "timecode", "timeSeconds", "outputPath", "width", "height", "method", "attempts", "succeededWithArgs"?, "presetPath"?, "fileSizeBytes"}` — exports a single PNG frame at the given `"MM:SS:FF"` timecode from the given (or active) sequence. Premiere writes under variant names (`frame.png` → `frame.png.png` from QE, `frame0.png` from AME) — both are detected by direct path checks (NOT directory enumeration, which macOS returns empty for e.g. `/tmp` — same TCC quirk as the `/Applications` note above) and renamed to the exact requested path. ## ExtendScript layout

Each command's implementation lives in its own file,
`host/commands/<command-name>.jsx` (e.g. `host/commands/export-frame.jsx`),
defining one `ppb_`-prefixed function. `host/index.jsx` — the manifest's
single `<ScriptPath>`, evaluated eagerly on panel load — holds only the
helpers shared by more than one command plus `ppb_dispatch()`, which
`$.evalFile`s a command's file into the same global ExtendScript context
the first time that command is called (and caches it for the rest of the
panel session; reloading the panel clears the cache, so the
edit-command-file → reload-panel dev loop picks up changes).

Adding a command takes three registrations:

1. `host/commands/<name>.jsx` — the `ppb_*` implementation
2. the `PPB_COMMANDS` map in `host/index.jsx`
3. the `ALLOWED_COMMANDS` list in `js/main.js`

## Notes

- No persistence: closing/reopening the panel clears the log.
- ExtendScript handles metadata queries (e.g. `get-project-info`) and
  structural operations like `create-sequence`. General timeline editing
  (clip trimming, track manipulation, rendering) is not supported.
- If port 47823 is already in use (e.g. a second panel instance), the panel
  shows an error in its own log view instead of failing to load.
